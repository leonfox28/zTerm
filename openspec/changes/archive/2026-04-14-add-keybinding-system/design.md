## Context

zTerm 当前已有终端标签、多分屏、设置页与基础工作台结构，但大多数动作仍由具体组件内的鼠标事件直接触发，例如 `TerminalTabs.tsx` 的新建终端/分屏按钮，以及 Activity Bar 的设置入口。现状适合快速迭代，但一旦引入快捷键，如果仍让每个组件各自监听 `keydown`，会很快出现命令分散、作用域冲突、终端输入被误拦截的问题。

本次设计需要在不引入复杂 VS Code 式命令服务/上下文系统的前提下，为 zTerm 提供一套轻量但可扩展的快捷键基础设施，优先覆盖 handoff 中明确的下一优先级：快捷键系统。约束包括：
- 继续使用当前 React + Zustand + CSS variables 架构，不新增重型依赖。
- 默认以 macOS 开发环境为先，同时保留跨平台表达方式。
- 终端输入区域是高冲突场景，不能为了支持全局快捷键而破坏 xterm.js 的正常按键透传。
- Claude 负责规划，后续实现由 Codex 执行，因此设计需要把命令入口、作用域和首批功能边界写清楚。

## Goals / Non-Goals

**Goals:**
- 建立统一的“命令 + 默认快捷键 + 作用域判断 + 执行器”模型，集中管理工作台快捷键。
- 让现有终端相关高频操作可以同时被鼠标入口和快捷键入口复用同一套动作实现。
- 明确哪些快捷键可以在终端聚焦时继续生效，哪些必须避让文本输入/终端原始输入场景。
- 为后续设置页、SSH 连接管理等功能复用同一机制预留扩展位。

**Non-Goals:**
- 不实现 VS Code 级别的完整 key chord/context key/when clause 系统。
- 不在本次设计中加入用户自定义快捷键编辑 UI。
- 不覆盖所有未来命令，只定义首批终端工作区必需动作。
- 不改变 xterm.js 自身已有的 shell 输入行为。

## Decisions

### 1. 采用 renderer 级集中式 keybinding registry，而不是组件内分散监听
- 决策：在 renderer 工作台高层注册单一 `keydown` 监听，维护一个静态 keybinding registry，将组合键解析后映射到命令 id，再调用集中 command handlers。
- 原因：当前动作散落在 `ActivityBar.tsx`、`TerminalTabs.tsx` 等组件中，若各处单独监听会导致重复判断焦点、冲突难排查。集中式 registry 更适合当前规模，也方便后续补充命令列表。
- 备选方案：
  - 组件级监听：实现快，但随着功能增长会形成重复逻辑。
  - 直接引入第三方快捷键库：能省部分解析工作，但会引入额外抽象，且当前需求仍然偏轻量。

### 2. 命令与 UI 触发入口解耦，鼠标与快捷键共用 command handlers
- 决策：把“新建终端 / 分屏 / 关闭标签 / 切换标签 / 打开设置”等动作抽成统一命令执行函数，组件按钮点击和快捷键都只触发命令，不直接操作细节。
- 原因：这样可以避免一份逻辑写在按钮里、另一份逻辑写在快捷键里；也让后续 context menu 或命令面板复用成为可能。
- 备选方案：
  - 快捷键直接调用 store：短期可行，但会和现有按钮实现重复，扩展成本高。

### 3. 作用域使用轻量布尔上下文，而不是完整 when-clause 表达式
- 决策：每条命令只声明少量可计算上下文，例如 `requiresTerminalTab`、`allowWhenTerminalFocused`、`disallowWhenTextInputFocused`，由高层在触发时统一判断。
- 原因：zTerm 当前还没有 VS Code 那样的 context key 系统，直接上表达式语言会过度设计。首批快捷键所需判断主要集中在：是否有激活标签、当前主视图是否为 terminal、当前焦点是否在文本输入/设置表单中、当前焦点是否在 xterm 区域。
- 备选方案：
  - 完整 when-clause：更灵活，但当前仓库没有支撑体系，复杂度过高。
  - 完全不做作用域：会导致在设置输入框或终端 shell 输入中误触发。

### 4. 终端聚焦时仅允许白名单命令抢占按键
- 决策：当焦点位于 xterm 终端区域时，只允许明确声明 `allowWhenTerminalFocused` 的命令继续生效，例如新建终端、分屏、切换标签；其余快捷键不拦截，继续交给终端/ shell。
- 原因：终端是按键密度最高的区域，若默认全局拦截会破坏 shell 交互。白名单策略更安全，也更接近实际需求。
- 备选方案：
  - 终端焦点下全禁用工作台快捷键：会让新建/切换等高频动作体验变差。
  - 终端焦点下默认全允许：误伤 shell 输入风险太高。

### 5. 默认快捷键以“首批必需动作”覆盖为主，并以显示名规范化
- 决策：首批命令包括：新建终端、关闭当前标签、切换上/下一个标签、水平/垂直分屏、打开设置、从设置返回终端。registry 同时保存规范化显示标签，方便未来在 UI 中展示。
- 原因：这些动作都已在当前产品中存在明确入口，且优先级最高；先覆盖它们，能验证整个系统链路是否合理。
- 备选方案：
  - 一次性覆盖所有潜在动作：需求边界不清，容易拖大 scope。

## Risks / Trade-offs

- [快捷键与浏览器/Electron 默认行为冲突] → 选择首批组合键时优先避开高风险系统级保留键，并把冲突留为后续可调整配置。
- [终端焦点判断不准确] → 使用事件 target 向上匹配终端容器与表单控件，先覆盖当前 DOM 结构中的明确区域，避免模糊推断。
- [命令抽取后导致现有按钮回归] → 鼠标入口改为调用同一 command handler，并保留现有 store 流程，减少行为漂移。
- [未来需要用户自定义快捷键时当前模型不够] → registry 结构预留 `commandId + defaultKeybinding` 形式，后续可在不推翻命令层的前提下接入持久化覆盖。

## Migration Plan

1. 新增 renderer 侧 keybinding/command 模块，定义命令表、组合键标准化与监听入口。
2. 将现有高频 UI 动作抽到 command handlers，并让按钮点击改为复用这些 handlers。
3. 在工作台高层挂载统一监听，根据当前 main view、活动 tab、焦点元素判断是否允许执行命令。
4. 为首批终端命令接入默认快捷键并补充静态校验。
5. 通过 lint、tsc 和用户手动 GUI 验证确认终端输入未被破坏、快捷键触发生效。

## Open Questions

- 首批默认快捷键的具体组合是否需要完全对齐 VS Code，还是只保持“接近且不冲突”的体验。
- 后续是否要把命令显示名与快捷键标签暴露到右键菜单或设置页中。
